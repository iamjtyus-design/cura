# CURA Product Constitution

## Document Purpose

This document is the primary source of truth for CURA.

Every product, design, engineering, privacy, marketing, pricing, and roadmap decision should be checked against this document before implementation.

When another document conflicts with this Constitution, this Constitution wins unless it is intentionally revised.

---

## 1. Product Identity

**Product name:** CURA  
**Parent company:** VisionBuilt  
**Name meaning:** Capture. Understand. Refine. Activate.  
**Primary promise:** Capture once. Create everything you need next.  
**Supporting line:** Turn captured moments into ready to use notes, content, visuals, and action.  
**Category:** Mobile multimodal capture and knowledge activation platform.

CURA is a capture product first.

It is not a professional editing suite, social scheduler, general purpose research notebook, or full business operating system.

CURA captures real world information, understands it, organizes it, and rapidly moves the user toward the next useful action.

---

## 2. VisionBuilt Ecosystem Role

CURA is the capture gateway into the VisionBuilt ecosystem.

The long term ecosystem is:

1. CURA captures and structures information.
2. The VisionBuilt knowledge layer connects ideas, people, projects, decisions, actions, and sources.
3. Mission Control visualizes and orchestrates that information.
4. Future VisionBuilt tools help users execute, automate, and build from it.

CURA must deliver strong standalone value before Mission Control exists.

CURA should never depend on a future dashboard to feel complete.

---

## 3. Product Philosophy

### Capture should be effortless

Users should be able to begin with audio, video, photos, screenshots, text, documents, websites, YouTube links, and supported social links.

### Understanding should be intelligent

CURA should produce more than raw transcription. It should identify meaning, context, themes, decisions, ideas, actions, visual information, and relationships.

### Organization should feel automatic

The user should receive a coherent curated note, not a pile of disconnected AI outputs.

### Output should be immediate

Users should quickly move from capture to something they can use, share, save, study, publish, or act on.

### Ownership should remain with the user

Users should be able to export their information to Files, iCloud Drive, Google Drive, Dropbox, or another user controlled location where supported.

### Motivation should be practical

CURA should create momentum without using exaggerated language.

The user should feel:

> I captured it. I understand it. I can do something with it now.

## 4. Core User

CURA is designed for the everyday capturer.

The primary user regularly encounters information worth keeping and wants help turning it into something useful.

Three universal modes organize the product:

### Learn

For church, school, lectures, conferences, books, sermons, research, training, podcasts, and educational videos.

### Create

For interviews, social media, content ideas, articles, newsletters, captions, carousels, quote posts, creator workflows, and faceless accounts.

### Work

For meetings, property walks, site visits, inspections, consulting sessions, project conversations, sales calls, interviews, and operational notes.

These modes are not separate products.

They are different output pathways powered by the same capture and understanding engine.

## 5. Core Jobs To Be Done

CURA helps users:

1. Capture something before it is forgotten.
2. Understand a long or complicated source quickly.
3. Turn a real world experience into organized notes.
4. Extract important ideas, decisions, tasks, and follow ups.
5. Create useful content from a captured moment.
6. Convert a lecture, sermon, or video into learning materials.
7. Preserve information in a personal knowledge system.
8. Move information into another app with minimal friction.
9. Reuse one capture for several different outcomes.
10. Build a structured knowledge history that can later feed Mission Control.

---

## 6. Canonical Product Flow

The core CURA flow is:

1. Capture
2. Understand
3. Refine
4. Activate

In product terms:

1. The user records, uploads, pastes, scans, photographs, or shares a source.
2. CURA processes audio, visuals, text, and metadata.
3. CURA creates one canonical Curated Note.
4. The Curated Note feeds mode specific outputs.
5. The user saves, copies, shares, exports, or opens another app.

CURA should not generate disconnected outputs directly from raw files when the Curated Note can serve as the shared source of truth.

## 7. Capture Session

The top level object in CURA is a Capture Session.

A Capture Session may include one or several sources from the same real world experience.

Examples:

1. An interview recording with photos and a website link.
2. A property walkthrough video with spoken observations and screenshots.
3. A church service recording with presentation slides.
4. A conference session with audio, notes, photos, and a speaker profile.
5. A meeting recording with a PDF agenda.
6. A YouTube video with typed personal notes.
7. A social media video with a related article.

A session should support:

1. One source or many sources.
2. Adding sources before or after processing.
3. Source specific outputs.
4. Combined session outputs.
5. Time aligned evidence where possible.
6. Detection of unrelated audio and visual tracks.
7. Clear source origin for every generated statement.

## 9. Multimodal Understanding

CURA must understand both the audio and the visual layer of a video.

For video, CURA should analyze:

1. Spoken audio.
2. Visible text.
3. Screens and interfaces.
4. Slides and diagrams.
5. Scene changes.
6. Objects and environments.
7. Actions demonstrated on screen.
8. Relationships between the visuals and the audio.

If the visual content and spoken content are unrelated, CURA should separate them.

Example:

1. Audio track: teaching about 1 Timothy 2.
2. Visual track: Apple Shortcuts workflow.
3. Relationship result: likely unrelated.
4. Output options: audio only, visual only, or combined session.

This multimodal behavior is a core differentiator.

## 16. Storage and Privacy Position

CURA should be privacy forward and honest.

### Default posture

1. Local first library where practical.
2. Originals remain on the user device unless processing requires temporary upload.
3. Server processing is temporary by default.
4. Temporary processing files are deleted after completion under a documented schedule.
5. Permanent CURA cloud storage is not enabled by default.
6. User controlled export is encouraged.
7. Future Mission Control cloud sync is explicit opt in.

### Modes

#### Private Mode

1. Local processing where supported.
2. Limited cloud AI capabilities.
3. No permanent CURA server storage.

#### Smart Mode

1. Temporary encrypted upload.
2. Cloud AI processing.
3. Results returned to the device.
4. Temporary files deleted automatically.
5. No model training on user content by default.

#### Future Sync Mode

1. Explicit opt in.
2. Persistent cloud storage.
3. Cross device access.
4. Mission Control integration.
5. Clear retention and deletion controls.

### Required user controls

1. Delete original.
2. Delete derived outputs.
3. Delete entire session.
4. Export data.
5. Review processing mode.
6. Choose backup location.
7. Disable analytics where possible.
8. Understand which vendors process content.

## 17. Legal and Safety Principles

CURA should minimize risk through product design, not disclaimers alone.

Required practices:

1. Operate through an appropriate legal entity.
2. Display clear recording consent language.
3. Never enable hidden recording.
4. Remind users that recording laws vary.
5. Require users to confirm rights to uploaded media.
6. Do not market CURA as a platform restriction bypass.
7. Do not train on user content by default.
8. Use truthful AI accuracy disclosures.
9. Label inferred information.
10. Avoid medical, legal, financial, or emergency reliance claims.
11. Provide deletion and export controls.
12. Maintain vendor agreements and security practices.
13. Carry cyber and errors and omissions insurance when commercially appropriate.
14. Obtain legal review before public launch.

Suggested recording notice:

> Only record people when you have permission and when recording is lawful. Consent requirements vary by location. You are responsible for notifying participants and obtaining any required consent.

1. Use a simple Poppins wordmark initially.

## 20. Technical Principles

1. Native SwiftUI for the iPhone app.
2. App Intents are a first class product surface.
3. AVFoundation handles recording.
4. Apple Vision may handle local OCR and frame text.
5. Gemini handles supported public YouTube and multimodal video understanding.
6. A dedicated speech to text provider handles verbatim audio transcription.
7. Supabase supports authentication, temporary processing coordination, billing state, and future sync.
8. RevenueCat manages subscriptions.
9. API keys remain server side.
10. Every AI output uses a structured schema.
11. Every output records model and prompt version.
12. Every important statement should preserve source evidence.
13. Processing should be asynchronous and retryable.
14. The app must recover from interrupted uploads and recordings.
15. Business logic should support the future web dashboard.
16. No platform specific shortcut should become the only place critical logic exists.

## 21. Apple Shortcuts Principles

Shortcuts are a core differentiator.

Priority actions include:

1. Record with CURA.
2. Process a file with CURA.
3. Analyze text with CURA.
4. Add source to a recent session.
5. Get latest Curated Note.
6. Generate an output pack.
7. Get open action items.
8. Export to Second Brain.
9. Find captures.
10. Ask CURA about a session.

Shortcuts should return structured data.

They should not silently send messages, publish content, create appointments, or take consequential actions without user confirmation.

1. Time saved.
2. Number and duration of captures.
3. Multimodal processing cost.
4. Premium output packs.
5. Translation.
6. Visual Brief generation.
7. Advanced Shortcuts.

A paid tier should support recurring use.

## 27. Decision Filter

Before adding any feature, ask:

1. Does it improve capture?
2. Does it improve understanding?
3. Does it improve the Curated Note?
4. Does it help the user activate the information?
5. Does it support Learn, Create, or Work?
6. Does it maintain privacy and ownership?
7. Does it preserve evidence?
8. Does it fit the mobile first experience?
9. Does it avoid turning CURA into an editor or scheduler?
10. Does it strengthen the future VisionBuilt ecosystem?

If the answer is no to most of these questions, the feature should not enter V1.

---

## 28. Final Product Definition

CURA is a mobile first, multimodal capture and knowledge activation platform by VisionBuilt.

It lets everyday people capture audio, video, photos, text, documents, web pages, YouTube videos, and supported social content.

CURA understands the complete experience, creates one organized Curated Note, and transforms it into useful outputs across Learn, Create, and Work.

Users can immediately save, copy, share, export, or move those outputs into the tools they already use.

CURA does not replace professional editors, publishing platforms, research notebooks, or Mission Control.

It is the trusted capture gateway that turns moments into momentum.
